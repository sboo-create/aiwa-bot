# AIWA · Аналитика 2.0 — спецификация метрик (на утверждение)

Цель: убрать «месс», дать корректные и однозначные метрики с точными определениями и формулами. После твоего ОК переделаю дашборд и сообщение `/stats` строго по этому документу.

Принципы: (1) у каждой метрики одно определение и одна формула; (2) одинаковые названия не значат разное; (3) «событие» и «вызов модели» — разные вещи, считаются отдельно; (4) числитель и знаменатель всегда явные.

---

## 1. Событийная модель — что реально логируется

Всё пишется в одну таблицу `events(chat_id, ts, action, tokens, meta, ms, n, calls)`. События идут **и из чата, и из приложения**; источник различается по `meta` (у приложения `web_*`, `view_*`, `app_open`).

| action | что это | инициатор | примеры meta |
|---|---|---|---|
| command | команда бота (/today, /menu) | пользователь (чат) | — |
| button | тап инлайн-кнопки или кнопки в аппе | пользователь (чат+апп) | app_open, view_food, web_diary, web_meal_replace |
| suggest | тап по подсказке-чипу | пользователь (чат+апп) | q:… |
| manual | набранное сообщение / ручное действие | пользователь (чат+апп) | web_checkin, food_log, workout, web_period_* |
| answered | Айва ответила в чате | пользователь → ответ | answer, general, partner_q |
| voice | голосовое сообщение | пользователь (чат+апп) | — |
| fallback | Айва не смогла ответить | пользователь → неудача | gibberish |
| error | ошибка / исключение | система | тип ошибки |
| signup / activated | регистрация / активация | система-маркер | — |
| goal | маркер целевого действия | система-маркер | food_log, workout, checkin, summary, report, partner_link |
| tokens | ГЕНЕРАЦИЯ моделью (несёт tokens и calls) | система (следствие запроса) | summary, menu, food_photo, food_text, meal, workout, diary_reco, answer |
| broadcast | системная рассылка / пуш | система | sent, blocked, error, checkin_push, food_reminder_sent, train_reminder_sent, phase_push, reactivation_sent, announce_sent |

Поля: `tokens` — токены; `calls` — **число реальных вызовов API модели** для события (фото = 3); `ms` — латентность; `n` — длина ввода.

---

## 2. Что сломано сейчас (диагноз)

1. **Тул-коллы и инференс-вызовы смешаны.** «tool_calls» = произвольный поднабор (`tokens || voice || answered+webapp`), а не «все обращения к модели». Рядом «llm_calls» = Σ calls. Метрики `req_per_dau` (в kpi) и `per_dau_req` (в throughput) названы похоже, но считаются от разных чисел → прямой путь к неверным выводам.
2. **Двойное логирование чат-ответа.** Один вопрос пишет и `answered`, и `tokens(meta=answer)`. При наивном подсчёте «обращений к модели» задваивается.
3. **DAU-набор действий не зафиксирован** — неясно, что считается активностью и не делает ли получение пуша пользователя «активным».
4. **Нет когортного и rolling-ретеншена, нет выгрузки, нет тултипов с определениями** — то, что ты просишь.

## 3. Слои метрик

### Слой 1. Аудитория

Активный день = уникальный chat_id с ≥1 ПОЛЬЗОВАТЕЛЬСКИМ событием. Набор пользовательских действий (единый для всех метрик активности):
`command, button, suggest, manual, answered, voice, fallback`.
НЕ входят: broadcast (это мы шлём), tokens (генерит система), goal/signup/activated/error (маркеры).

| Метрика | Определение | Формула |
|---|---|---|
| DAU | уникальные активные сегодня | count(distinct cid, day=today) |
| WAU | уникальные активные за 7 дней (скользящее) | count(distinct cid, day ∈ [today-6; today]) |
| MAU | уникальные активные за 30 дней | count(distinct cid, day ∈ [today-29; today]) |
| Средний DAU | среднесуточные за период | Σ(дневных DAU) / число дней |
| Средний WAU/MAU | среднее скользящих окон по дням | mean(WAU_d) / mean(MAU_d) |
| Stickiness | липкость (в точке и графиком) | DAU / MAU × 100% |

Ретеншен (когортный, по дню первой активности):

| Метрика | Определение | Формула |
|---|---|---|
| Retention D1 | доля новых, вернувшихся на 1-й день после первого | active_on(day0+1) / размер_когорты |
| Retention D7 / D30 | то же для 7 / 30 дня | аналогично |
| Rolling retention DN | доля новых, активных на N-й день ИЛИ позже | active_on(≥day0+N) / размер_когорты (устойчивее к пропускам) |

Сегментация: по режиму (цикл / планирование / беременность / менопауза / без цикла) — доля и активность; партнёры (подключено, у скольких женщин, уникальных); график «новые vs вернувшиеся».

### Слой 2. Вовлечённость (события и модель)

Разделяем ТРИ разных числа — это ядро наведения порядка:

| Метрика | Что считает | Формула | 1 фото еды |
|---|---|---|---|
| Событий на DAU | все пользовательские действия (чат+апп) | Σ польз. событий / активные·дни | несколько |
| Обращений к ИИ на DAU (тул-коллы) | сколько раз пользователь запустил ИИ-функцию | число запросов к модели / активные·дни | 1 |
| Инференс-вызовов на DAU | реальные вызовы API модели (нагрузка/стоимость) | Σ calls / активные·дни | 3 |

«Активные·дни» (знаменатель) = Σ по дням числа активных за период (person-days) — корректный знаменатель для «на DAU».

Дополнительно: распределение действий по типам и по источнику (чат vs приложение); сессии (пауза >30 мин = новая): сессий/DAU, средняя длина, действий/сессия.

### Слой 3. Продукт

| Метрика | Определение |
|---|---|
| Рассылки | отправлено / заблокировали / ошибок по типам (сводка, чек-ин, пуш еда, пуш тренировка, фаза, реактивация, анонс) |
| Конверсия пуш → открытие | доля получивших пуш, кто открыл аппу/ответил в тот же день |
| Воронка активации | signup → онбординг → первая сводка → первый лог еды → первая тренировка |
| Adoption по фичам | сколько активных пользуются дневником еды / тренировками / чатом / партнёром |

### Слой 4. Качество ответов

| Метрика | Определение | Формула |
|---|---|---|
| Успешность ответов | доля успешных из попыток | answered / (answered + fallback + error) × 100% |
| Доля фолбэков | не смогла ответить | fallback / попытки × 100% |
| Доля ошибок | сбои | error / попытки × 100% |
| Латентность | скорость ответа | p50 / p95 по ms события answered |
| Токены и стоимость | нагрузка на тариф | Σ tokens; стоимость = Σ tokens / 1e6 × цена (env) |

## 4. Ключевой нюанс: тул-колл ≠ инференс-вызов

- Тул-колл (обращение к ИИ-функции) — ПРОДУКТОВАЯ метрика: сколько раз пользователь воспользовался умной функцией (фото, вопрос в чате, обновление меню). Одно фото = 1 тул-колл.
- Инференс-вызов — ТЕХНИЧЕСКАЯ метрика нагрузки: каждый реальный вызов модели. Одно фото = 3 (ввод/распознавание/вывод), т.к. calls = len(usage) фиксирует под-вызовы.

Обе нужны для разных целей: тул-коллы — ценность и вовлечённость (твоя KPI), инференс — нагрузка и стоимость. В дашборде — ЯВНО разные строки с разными подписями, никогда под одним именем.

---

## 5. Обязательные правки логирования (чтобы считалось точно)

1. Убрать двойное логирование чат-ответа: сейчас пишутся и `answered`, и `tokens(meta=answer)`. Оставить один канонический маркер «обращение к модели» на запрос (с calls), а `answered` — только для латентности/успешности (без calls). Тогда тул-коллы считаются без задвоения.
2. Зафиксировать DAU-набор одной константой, чтобы все метрики активности считались от одного набора.
3. (опц.) Явный флаг источника (чат/приложение) в meta по единому правилу.

---

## 6. Формат вывода

Дашборд (веб, по ключу): вкладки Аудитория · Вовлечённость · Продукт · Качество.
- Каждая ключевая метрика: значение в точке + график динамики; наведение на точку → число за день; наведение на значок ⓘ → определение и формула.
- Выгрузка в Excel — сырой ряд по дням (DAU/WAU/MAU, события, тул-коллы, инференс, рассылки, качество).
- DAU/WAU/MAU, stickiness, retention, события/тул-коллы/инференс на DAU — графиками; сегментация и распределения — таблицами/барами.

Сообщение `/stats`: компактная выжимка теми же четырьмя блоками; каждая строка с явными числителем и знаменателем (напр. «Событий на DAU: 6.3 = 1020 событий / 162 активных·дня»).

## 7. Что перепроверю перед публикацией

- Синтетические тесты на каждую формулу (посевная БД с известным ответом).
- Сверка события/DAU vs тул-коллы/DAU vs инференс/DAU на одних данных — фото даёт 1 тул-колл и 3 инференса.
- Ретеншен на ручной когорте.
- Что получение пуша НЕ делает пользователя активным.
