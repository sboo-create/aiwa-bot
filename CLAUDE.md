# AIWA — контекст проекта для Claude Code

AIWA (AI for Woman Awareness, «Айва») — бесплатный проактивный ИИ-консультант по женскому здоровью
на русском языке: Telegram-бот + мини-приложение (Telegram Mini App).
Покрывает цикл, беременность, менопаузу, нерегулярный цикл; питание, нагрузку, самочувствие.

Текущая версия: `AIWA_VERSION = "2026-07-13-v51"` (константа в `aiwa_bot.py`, строка ~85).

---

## 1. Стек и архитектура

- Python 3.12, один asyncio-процесс: `python-telegram-bot[job-queue]==22.8` + `aiohttp` веб-сервер в том же loop.
- Хранилище: SQLite (WAL), путь из `AIWA_DB`.
- Деплой: Railway, `Procfile` → `worker: python aiwa_bot.py`.
- LLM: GigaChat. Основной путь — LiteLLM-прокси в OpenAI-совместимом формате; есть фолбэки
  (прямой GigaChat API и «стенд»). Выбор движка — `AIWA_PROVIDER`.
- STT (голосовые): Groq Whisper (`GROQ_API_KEY`) — это рабочая зависимость, не легаси.
- Vision (распознавание еды по фото): GigaChat Vision.

Бот и мини-апп живут в одном процессе: aiohttp отдаёт `webapp/index.html` и все `/api/*` эндпоинты,
а PTB держит polling. Мини-апп авторизуется через Telegram `initData` (проверка подписи в `_verify_init`).

## 3. КРИТИЧЕСКИ ВАЖНО: порядок проверки после любой правки

Это главное правило проекта. Мини-апп существует в двух файлах, и рассинхрон ломает прод молча.

```bash
# 1. Python компилируется
python3 -m py_compile aiwa_bot.py llm.py

# 2. JS мини-аппа валиден (вытащить <script> и проверить)
python3 -c "
import re; h=open('aiwa_webapp.html',encoding='utf-8').read()
open('/tmp/app.js','w',encoding='utf-8').write('\n'.join(re.findall(r'<script>(.*?)</script>',h,re.S)))"
node --check /tmp/app.js        # расширение .js обязательно, иначе ERR_UNKNOWN_FILE_EXTENSION

# 3. Синхронизировать копию — ОБЯЗАТЕЛЬНО после каждой правки html
cp aiwa_webapp.html webapp/index.html
diff -q aiwa_webapp.html webapp/index.html   # должно быть «идентичны»

# 4. Поднять версию AIWA_VERSION в aiwa_bot.py (иначе не понять, что доехало)
```

Прочие соглашения:
- Правки в больших файлах делать точечной заменой уникальной строки с проверкой, что она встречается ровно один раз.
  Файл записывать только после того, как все проверки прошли.
- Админ-дашборд собирается как `html_text = r"""..."""` (raw-строка, НЕ f-string) — фигурные скобки в JS не экранировать.
- Если дашборд «пустой» — почти всегда кэш браузера, проверять с Ctrl+Shift+R.
- Деплой делает владелец через `git push`. Не деплоить без явного согласия.

## 4. Данные (таблицы SQLite)

`users` (профиль, режим, время сводки, kcal_goal, diet_note), `cycles`, `intimacy`, `logs` (симптомы,
энергия, настроение по дням), `meals` (дневник питания), `workouts` (тренировки), `chat_log`, `sugg`,
`partners`, `events` (аналитика), `proactive_log`, `proactive_state`, `memory` (долгая память),
`referrals` (источники переходов).

Событие аналитики: `ev(cid, action, tokens, meta, ms, n, calls)`.
`calls` = число хопов к модели для этого события; сумма `calls` = тул-коллы в дашборде.
Глобальный `STATS["calls"]` в `llm._call` — источник правды по общему числу вызовов модели.

## 5. Слой модели (`llm.py`)

- `_call(messages, max_tokens, temperature, usage, attempts)` — единая точка вызова, считает статистику.
- `_clean(out, fallback)` — постобработка ответа.
- `call_tools(messages, tools, usage, ...)` — раунд OpenAI function-calling. Возвращает `None`,
  если провайдер не поддерживает tools → вызывающий код должен корректно откатываться.
- Генерации: `generate_summary`, `answer_question`, `general_answer`, `menu_today`, `training_today`,
  `food_suggestions`, `today_note`, `proactive_compose`, `practice_reflection`, `memory_extract`,
  `analyze_food` / `analyze_food_text` (Vision), `diary_reco`, `partner_*`, `transcribe` (Groq).

### Агент (в `aiwa_bot.py`)
Двухфазная схема, чтобы tool-use не ухудшал качество ответа:
1. `_agent_answer` — модель через `call_tools` сама добывает реальные данные пользовательницы.
   Инструменты (`_agent_tools_spec` / `_agent_exec`): `cycle_status`, `recent_symptoms`, `today_diary`,
   `recent_workouts`, `user_profile`, `recall`, `remember`.
2. `_agent_final` — финальный ответ пишется «богатым» промптом (`answer_question` / `general_answer`)
   с подмешанными данными и долгой памятью.
Если агент падает или tools не поддержаны — `_chat_reply` откатывается на обычный ответ.

### Долгая память
`memory(chat_id, mkey, mval, updated)`, максимум 40 записей на пользователя.
Помимо инструментов `remember`/`recall`, после каждого ответа в фоне запускается `_memory_learn`
(`llm.memory_extract`) — вытаскивает устойчивые факты. Работает даже если tools не поддерживаются.
`_with_memory(cid, q)` подмешивает память в финальные промпты. UI памяти намеренно нет.

## 6. Мини-апп (`aiwa_webapp.html`)

Вкладки: `today`, `food`, `train`, `stats`, `chat`. Переключение — `go(tab)`, старт — `boot()`.
`boot()` читает query-параметр `?tab=` и открывает нужную вкладку (используется проактивными кнопками).

Ключевые функции: `renderToday`, `renderFood` + `diaryHTML`/`renderDiaryBox`, `renderTrain` + `renderTrainLog`,
`renderStats`, `renderHub` (профиль-шторка), практика дыхания (`openPractice`, `startBreath`, SVG-цветок + кольцо).

Профиль (`renderHub`): «Мои данные» — рост/вес/возраст/длина цикла; «Предпочтения по питанию» —
диет-заметка **и калораж** (поле желаемых калорий + пресеты Дефицит/Норма/Профицит).
Сохранение: `saveHubProfile` → `/api/profile`, `saveHubPrefs` → `/api/prefs` (диета + kcal_goal).

Расчёт калорий (`calc_calories` в `aiwa_bot.py`): Миффлин–Сан Жеор для женщин
`BMR = 10*вес + 6.25*рост − 5*возраст − 161`, далее `TDEE = BMR × коэффициент активности`
(1:1.2, 2:1.375, 3:1.55, 4:1.725, 5:1.9). Дефолт активности — 2. Белок 1.6 г/кг, жиры 30% калорий,
остальное углеводы. Формула проверена: для 158 см / 48 кг / 31 год BMR = 1152.

## 7. Пуши и проактив

Джобы регистрируются в `main()`. Проактив включается флагом `AIWA_PROACTIVE`.

**Когда проактив включён, все прочие фоновые пуши отключаются** (еда, тренировки, смена фазы,
реактивация) — остаётся только утренняя сводка и максимум одно проактивное сообщение в день.
Это сделано намеренно, чтобы не выжигать пользователей.

Проактив: `_proactive_signals` (сигналы `felt_bad`, `pms_soon`, `low_energy`, `no_move`, `streak_N`,
`low_protein`, `practice_eve` со скорами и кулдаунами) → выбирается лучший → `L.proactive_compose` пишет
текст → отправка с клавиатурой. Дедуп через `proactive_state`, дневной лимит через `proactive_log`.
Тики в 13:00 и 19:30 МСК. Сухой прогон — админ-команда `/proactive`.

Кнопки проактивного сообщения:
- `_PA_ACTION` — саджест-кнопка («Да, собери тренировку» / «Что съесть на ужин?»), колбэк `pado:*`
  обрабатывается в начале `on_cb` и отвечает через `_chat_reply` реальной тренировкой/меню прямо в чате.
- `_PA_TAB` / `_pa_deeplink` — кнопка «Открыть Айву» ведёт сразу на нужную вкладку (`?tab=train|food`).

Важное правило по тону: проактивные сообщения не должны обещать, что Айва сделает что-то сама
автоматически, если по кнопке это не произойдёт. Промпт `proactive_compose` это уже учитывает.

## 10. Ключевые переменные окружения

Обязательные: `BOT_TOKEN`, `AIWA_WEBAPP_URL`, `AIWA_DB`, `AIWA_ADMIN`, `GROQ_API_KEY` (голос),
LLM-доступы (`LITELLM_URL` / `LITELLM_KEY` / `LITELLM_MODEL` либо GigaChat-креды), `AIWA_PROVIDER`.

Проактив: `AIWA_PROACTIVE=1`, `AIWA_PROACTIVE_SHADOW=0` (боевой режим), `AIWA_PROACTIVE_IDS`
(пусто или `all` = все пользователи), `AIWA_PROACTIVE_MIN` (порог скора), `AIWA_PROACTIVE_DELAY`.

Прочее: `AIWA_TZ` (по умолчанию Europe/Moscow), тайминги пушей (`AIWA_FOOD_PUSH_TIME` и т. п. —
не действуют при включённом проактиве), воркеры и лимиты рассылки.

## 11. Что сейчас в работе (бэклог по фидбеку тестировщиц)

**Баги (приоритет):**
- Рекомендованное меню не попадает в рассчитанное КБЖУ (пример: норма 1739 ккал, меню на 1260) — критично.
- Не сохраняются данные в разделе «Чем занимаешься» на вкладке «Нагрузка».
- «Путаница в днях» на «Нагрузке» — нужен повтор.
- В онбординге «при нажатии на кнопку ничего не происходит» — нужен повтор, неясно какая кнопка.

**Контент и безопасность (важно):**
- Добавить во все ответы дисклеймер, что дозировку и форму БАДов (магний, B6) подбирает врач;
  не давать конкретных назначений.
- Переписать формулировку про ПМС: из списка «тёмный шоколад, орехи, бананы, рыба, гречка»
  к сложным углеводам относится только гречка. Заменить на «продукты», добавить корректный список,
  по возможности сноски на исследования.
- Ответы сделать структурными: короткая суть сверху, детали ниже (жалобы на простыни текста).

**Улучшения:**
- Крестик для закрытия диалога с Айвой на вкладке «Сегодня» (в практике он есть, в чате нет).
- Граммовка блюд в меню.
- Порядок блоков на «Питании»: ручной ввод и КБЖУ выше, меню ниже.
- Быстрый переход из «Питания» в профиль для правки КБЖУ.
- Мультивыбор нагрузки (силовая + кардио одновременно).
- «Месячные» → «менструальный цикл» в заголовках.
- Отдельная быстрая тренировка на 10 минут как самостоятельная сущность.

**Позже (крупное):**
- Доп. метрики питания (сахар, клетчатка).
- Синхронизация с Apple Health / Google Fit (реалистично через агрегатор вроде Terra API,
  напрямую из Mini App HealthKit недоступен).

**Решено намеренно (не «баги»):**
- Онбординг принимает и `25.05.2026`, и `25 мая 2026` — намеренная гибкость.
- День цикла остаётся первичным, а не «сколько дней до менструации»: многие отслеживают овуляцию
  и конкретные дни цикла.
- Подробные ответы — намеренно (чувствительная тема), правим только структуру, а не глубину.
- Редактирование КБЖУ живёт в профиле, а не на вкладке «Питание».
- UI долгой памяти («что Айва про тебя поняла») намеренно убран.
