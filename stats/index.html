Как читать метрики, платформы и ошибки
Ever used — уникальные псевдонимные пользователи с продуктовой активностью за всё время.
… / user-day — итог периода, делённый на сумму активных пользователей за каждый день с данными.
Tools / DAU в Overview — все фактические AI-вызовы с 00:00 МСК, делённые на DAU той же московской даты. Успешные и ошибочные попытки, retry и fallback считаются: ошибка влияет на качество, но не отменяет факт вызова.
Actual tool execution — более узкий структурированный function call модели. Он показан отдельно и не заменяет общий продуктовый Tools / DAU.
Tool-assisted outcome — инструмент успешно вернул данные, после чего AIWA сформировала итоговый ответ. Это не заменяет пользовательскую оценку «Полезно».
Logical request — пользовательская или автоматическая задача с единым request_id. Один request может породить несколько provider attempts: первичный вызов, retry того же маршрута и fallback на другой маршрут.
Provider attempt — отдельное фактическое обращение к AI endpoint. Поэтому attempts больше requests. Внутренние действия рендера, SQL и кэша сюда не входят.
Request success — запрос успешен, если хотя бы один attempt с тем же request_id завершился success/ok/completed. Формула: successful requests / все requests при покрытии request_id от 80%.
Attempt error rate — неуспешные provider attempts / все provider attempts. Высокое значение означает лишнюю задержку и нагрузку даже при 100% Request success.
Recovered request — сначала была ошибка, затем retry или fallback того же request_id завершился успешно. Ответ доставлен, но путь был деградированным.
Terminal failure — ни одна попытка request не завершилась успешно. Это пользовательски значимая ошибка и главный аварийный сигнал.
Failure status — техническая причина attempt: HTTP 401/403 означает авторизацию, 429 — rate limit, 5xx — сбой upstream, timeout — превышение времени, transport_or_unknown — сетевое или неклассифицированное исключение.
Failure purpose — продуктовый сценарий, во время которого произошла ошибка: ответ в чате, сводка, генерация меню, распознавание голоса и т. п. Тексты сообщений и медицинские данные не сохраняются.
Push delivery — уникальная пара псевдонимного пользователя и campaign_id. Восемь неуспешных API-вызовов одной кампании считаются одной недоставкой и восемью техническими attempts.
Push blocked — Telegram вернул постоянный запрет доставки. Получатель исключается из фоновых очередей; новый входящий апдейт автоматически снимает suppression.
Push recovered — после ошибки та же user × campaign была успешно отправлена. Такая кампания не остаётся в терминальных недоставках.
Платформа — поверхность продукта: Telegram-бот или Mini App. Пользователи могут пересекаться. ОС устройства Telegram не передаёт в текущий privacy-safe слой, поэтому iOS/Android/Desktop не показываются.
Fallback request — request со сменой провайдера или повторной попыткой. Сам fallback полезен как страховка, но маршрут, который стабильно падает перед каждым успехом, нужно временно отключать circuit breaker.
Точно записанные (observed) — события, которые новая аналитика v2 записала напрямую. На них можно надёжнее смотреть воронки, токены и стоимость.
Восстановленные (reconstructed) — события, собранные из старой таблицы без сообщений и медицинских данных. Они расширяют историю, но могут быть неполными или менее точно классифицированными.
Стоимость — только вызовы с точной ценой от провайдера или точной оценки; стоимость по старой истории не восстанавливается.
